Intro

Welcome to PacTrac! We hope you enjoy our software. So what is PacTrac? PacTrac aims to provide a middleground SDK that analyzes emotional data from multiple input sources, such as sweat-detecting wristbands, EPOC headsets, and webcams. PacTrac takes in data from these sources and analyzes each set of data, causing a developer to receive one uniform set of data for later use, such as tracking most-popular apps on a computer from multiple sources or a desktop that personalizes itself to the mood given. And the best part? All of the processing is done server-side. All you need to do is send a JSON array, and we'll store the data, analyze it with previous data and current data, and return requests that you send within the JSON. It's that simple. Or, if you feel uncomfortable storing your data on our side of the server, you can download the free server-side code on our repositories.

Creating an account

Creating an account is free and simple. All you need to do is give us your email, password, first name and last name. There is only one account per email, though, due to litmitations with storage. However, each user is allowed to have five applications on his or her account, with unlimited clients for each application.


Directions for creating an account
  1. Go up to the top of the page.
  2. Click on the "Sign up" button
  3. Enter the appropriate information in each field
  4. After being brought to a "Registration Confirmed" page, you'll have an account!
  5. Sign in at the top of the page (no email confirmation, don't worry)

Once you've created your account, you want to go ahead and create applications. Each application will have its name and UUID (we'll explain this later) displayed in your home page.

How to create an app
  1. Enter the name of your app.
  2. Click "Create app"

Yep. It's that simple to create an app. Don't worry, we'll keep track of the amount of apps you have, so you can't go overboard.

Downloading packages for each operating system

So you're probably wondering, "If this is all server side, why should I need to download a package?" The packages we've created are meant to expidite and optimize your packaging systems. We've created a Mac package so far, which should be easily integratible into your application. You shouldn't need to do a ton of work to unify all of the data you're trying to send us.

Mac Packages

Mac Packages can be downloaded on our github repository. The client-side code is stroed in an Xcode Project filetype. So, our framework should work out of the box if you just drag and drop our project into your application.

Windows Packages

We don't currently have any helper Windows development files, but you can still send us data through a JSON request.

Linux Packages

Linux packages are in development, but for the time-being, it looks like you guys are in the same boat as the Windows developers (see above). Sorry.

Integrating with your app

Integration takes little to no work, solely because all you need to do is put the data you're receiving in a JSON array and ship it to http://www.pac.trac/detect.json. We'll automatically ship back the results you ask for once we compute them.

Integrating with your Mac app
Without our package

If you're not using our wrapper class, then we would recommend using either ASIHTTPRequest or MKNetworkKit to send and receive data. We would recommend using SBJSON to package and receive JSON requests. Just send us your data, and we'll send you back results.

Without package

If you are using our package, you can just drag-and-drop our file into your project. There should be zero configuration necessary to run our SDK. And it's all open-source!

x86 vs. x86-64

Our Xcode project file is configured to run only in x86-64. But this can easily be changed to run in x86.

  1. Go to the PacTrac Xcode Project File.
  2. Click on the "Build Settings"
  3. Find the "Deployment Architecture" setting
  4. Change it to x86

Lifecycle of a sample application

The lifecycle of what you need to implement is actually quite simple. Gather your data, send it to our server, and we return the data. This section will discuss this process in a little more depth.


Lifecycle of an app running for the first time
  1. The client sends a request to the server
  2. The server checks if the JSON array is valid
  3. The server checks if the client is new or existing, using a UUID that the client saves (we promise, it's the ONLY thing the client needs to save)
  4. Because the client is new, the server sends back a UUID
  5. The client saves the UUID (if you're not using our packages, then you need to do this manually)
  6. The client sends another request
  7. The server checks if the JSON array is valid
  8. The server sees the UUID now
  9. The server processes the data by parsing whatever data it gets from the JSON array sent to it
  10. The server saves the data
  11. The server sends back any info that the user requested
Lifecycle of a previously used app
  1. The client sends a request to the server
  2. The server checks if the JSON array is valid
  3. The server checks if the client is new or existing, using a UUID that the client saves (we promise, it's the ONLY thing the client needs to save)
  4. The server processes the data by parsing whatever data it gets from the JSON array sent to it
  5. The server saves the data
  6. The server sends back any info that the user requested
